# Multi-Stage Dockerfile — Dev / Prod Lockfile Strategy

Universal pattern for writing Dockerfiles that support two dependency installation modes via **named stages** selected at build time with `--target`.

## Core Principle

| Stage | Behavior | Selected by |
|:---|:---|:---|
| `development` (+ `development-runtime`) | Free `install` (no lockfile required). Generates the lockfile inside the image. Retrieved on host via `make lock`. | `docker compose build` (compose.yaml `target: development-runtime`) |
| `production` (+ `production-runtime`) | Strict frozen install. Consumes the committed lockfile (`bun.lock.docker`). | CI pipeline: `docker build --target production-runtime` |

**No build args.** The stage selection is done via `target` in compose.yaml (dev) and `--target` in CI (prod).

## Essential Rules

> **Never use `--frozen-lockfile` in the development stage.**
>
> The dev stage runs a free install — the lockfile is a *product* of this stage, not an input.

> **The lockfile is always committed in the repo.** Production builds depend on it.

> **Never launch an ephemeral container (`docker run --rm …`) to generate the lockfile.**
>
> Use `make lock` (`docker create` + `docker cp`).

> **In a monorepo with turbo prune, never rely on the turbo-pruned lockfile for `--frozen-lockfile`.**
>
> Turbo prune generates an incomplete/corrupted lockfile for bun. The valid lockfile is the one produced by `bun install` in the `development` stage.

## Dockerfile Pattern — Monorepo with Turbo Prune

```dockerfile
# ── pruner: workspace subgraph ───────────────────────────────────────────────
FROM oven/bun:1-alpine AS pruner
WORKDIR /repo
COPY . .
RUN bun x turbo@2 prune @scope/my-app --docker --out-dir /pruned

# ── development: free install (generates lockfile) ───────────────────────────
FROM oven/bun:1-alpine AS development
WORKDIR /build

COPY --from=pruner /pruned/json/ ./
RUN bun install --linker hoisted

COPY --from=pruner /pruned/full/ ./
RUN cd packages/shared-lib && bun run build
# ... build other workspace packages ...

# ── development-runtime ──────────────────────────────────────────────────────
FROM node:24-alpine AS development-runtime
COPY --from=development /usr/local/bin/bun /usr/local/bin/bun
COPY --from=development /build/node_modules /node_modules
WORKDIR /app
COPY --from=development /build/apps/my-app/node_modules ./node_modules
COPY --from=development /build/apps/my-app/ ./
COPY --from=development /build/bun.lock ./bun.lock.docker
COPY .docker/my-app/entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

EXPOSE 3000
ENTRYPOINT ["/entrypoint.sh"]
CMD ["bun", "run", "dev"]

# ── production: strict install (consumes committed lockfile) ─────────────────
FROM oven/bun:1-alpine AS production
WORKDIR /build

COPY --from=pruner /pruned/json/ ./
COPY apps/my-app/bun.lock.docker ./bun.lock
RUN bun install --frozen-lockfile --linker hoisted

COPY --from=pruner /pruned/full/ ./
RUN cd packages/shared-lib && bun run build
# ... build other workspace packages ...

# ── production-runtime ───────────────────────────────────────────────────────
FROM node:24-alpine AS production-runtime
COPY --from=production /usr/local/bin/bun /usr/local/bin/bun
COPY --from=production /build/node_modules /node_modules
WORKDIR /app
COPY --from=production /build/apps/my-app/node_modules ./node_modules
COPY --from=production /build/apps/my-app/ ./
COPY .docker/my-app/entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

EXPOSE 3000
ENTRYPOINT ["/entrypoint.sh"]
CMD ["bun", "run", "start"]
```

### Key Points

1. **`development-runtime` copies `bun.lock` from the development stage** → `make lock` can extract it.
2. **`production` uses `COPY apps/my-app/bun.lock.docker ./bun.lock`** — from the build context (host), the file previously generated by `make lock` and committed.
3. **`production-runtime` has `CMD ["bun", "run", "start"]`** (built app), while `development-runtime` has `CMD ["bun", "run", "dev"]` (watch mode).
4. **No `ARG`** — stage selection is purely via `--target`.

## compose.yaml (local / dev)

```yaml
services:
  my-service:
    build:
      context: .
      dockerfile: .docker/my-service/Dockerfile
      target: development-runtime
```

Le compose cible **toujours** le stage `development-runtime`. Le stage `production-runtime` n'est jamais buildé via compose — il est construit par la CI avec `--target production-runtime`.

## Makefile — `make lock`

`make lock` builde via compose (qui cible `development-runtime`), crée un container éphémère pour extraire le lockfile, et le copie vers l'host.

```makefile
DOCKER  := docker
COMPOSE := $(DOCKER) compose

lock:
	@for app in auth-api app-registry entitlements feature-flags hub-shell; do \
		echo "  → Generating pruned lockfile for $$app..."; \
		$(COMPOSE) build $$app; \
		container_id=$$($(DOCKER) create modelo-hub-$$app); \
		$(DOCKER) cp $$container_id:/app/bun.lock.docker ./apps/$$app/bun.lock.docker; \
		$(DOCKER) rm $$container_id > /dev/null; \
		echo "  ✓ apps/$$app/bun.lock.docker updated"; \
	done
	@echo "✅  All pruned lockfiles generated. Commit them."

lock-%:
	@echo "  → Generating pruned lockfile for $*..."
	@$(COMPOSE) build $*
	@container_id=$$($(DOCKER) create modelo-hub-$*); \
	$(DOCKER) cp $$container_id:/app/bun.lock.docker ./apps/$*/bun.lock.docker; \
	$(DOCKER) rm $$container_id > /dev/null
	@echo "  ✓ apps/$*/bun.lock.docker updated"
```

### Why `docker create` + `docker cp` (not `docker compose cp`)

- `docker compose cp` requires a **running** container
- `docker create` creates a container from the image without starting it
- Allows extracting files without running the entrypoint (migrations, seed, etc.)

## CI / Production Build

```bash
docker build --target production-runtime \
  -f .docker/app-registry/Dockerfile \
  -t my-registry/app-registry:${TAG} .
```

## Complete Workflow

```bash
# 1. Modify package.json (add/remove a dependency)
vim apps/my-app/package.json

# 2. Build the dev image (compose target: development-runtime)
docker compose up -d --build my-app

# 3. Extract the lockfile generated by the development stage
make lock-my-app

# 4. Commit the lockfile
git add apps/my-app/bun.lock.docker
git commit -m "chore(my-app): update pruned lockfile"

# 5. CI/prod builds use --target production-runtime (consumes bun.lock.docker)
```

## Dockerfile Pattern — Simple App (No Turbo Prune)

For single-package apps without monorepo pruning:

```dockerfile
# ── development ──────────────────────────────────────────────────────────────
FROM oven/bun:1 AS development
WORKDIR /app
COPY package.json ./
RUN bun install

# ── production ───────────────────────────────────────────────────────────────
FROM oven/bun:1 AS production
WORKDIR /app
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
```

compose.yaml: `target: development`. CI: `--target production`.

## General Best Practices

### Multi-Stage Structure
- Use meaningful stage names with `AS`
- Order: pruner → development → development-runtime → production → production-runtime
- Copy only necessary artifacts to runtime stages

### Base Images
- Specify exact version tags (e.g., `oven/bun:1.3-alpine`)
- Use Alpine-based images for smaller footprints
- Use different images for build (bun) vs runtime (node) when appropriate

### Security
- Avoid running containers as root — use `USER` instruction
- Multi-stage builds naturally exclude build tools from final image
- Never include build secrets in the final image

## Checklist

```
[ ] Dockerfile: stage development — free install, no lockfile input
[ ] Dockerfile: stage development-runtime — copies bun.lock.docker for extraction
[ ] Dockerfile: stage production — COPY bun.lock.docker as bun.lock, --frozen-lockfile
[ ] Dockerfile: stage production-runtime — CMD start (not dev)
[ ] Dockerfile: NO ARG for target selection
[ ] compose.yaml: target: development-runtime
[ ] Makefile: target lock — build, docker create, docker cp, docker rm
[ ] apps/<app>/bun.lock.docker committed in the repo
[ ] CI: docker build --target production-runtime
```

## Anti-patterns

| Anti-pattern | Why it's wrong | Correct approach |
|:---|:---|:---|
| Using `--frozen-lockfile` in the development stage | The lockfile doesn't exist yet — this IS the stage that generates it | Only `--frozen-lockfile` in `production` |
| Using the turbo-pruned `bun.lock` for `--frozen-lockfile` | Turbo prune generates an incomplete/corrupted lockfile for bun | Use `bun.lock.docker` generated by `make lock` from development stage |
| Copying the full monorepo lockfile for frozen install after turbo prune | Full lockfile references packages absent from the pruned subset | Use `bun.lock.docker` from the pruned dev install |
| Using `ARG TARGET` to switch between dev/prod deps | Adds complexity, harder to read | Two separate stage chains, selected by `--target` |
| Running `docker run --rm` to generate the lockfile | Runs entrypoint (migrations), fragile | `docker create` + `docker cp` |
| Using `docker compose cp` in make lock | Requires a running container | `docker create` + `docker cp` (no start) |
| Not committing `bun.lock.docker` | Prod builds will fail | Always committed |
| `docker compose restart` after `.env` change | Does not re-read `.env` | `docker compose up -d <service>` (recreation) |
| Same `CMD` for dev and prod | Dev needs watch mode, prod needs optimized start | `CMD ["bun", "run", "dev"]` vs `CMD ["bun", "run", "start"]` |

## Troubleshooting

### `lockfile had changes, but lockfile is frozen`

**In production stage**: The `bun.lock.docker` is stale. Run `make lock-<app>` to regenerate, then commit.

### `Cannot find package '@scope/pkg'` at runtime

Volume anonyme Docker (`/app/node_modules`) stale. Rebuild:
```bash
docker compose down <service>
docker compose up -d --build <service>
```

### `turbo prune` generates corrupted lockfile

Known issue with turbo + bun. Never use turbo's pruned lockfile with `--frozen-lockfile`. Use the `make lock` pattern instead.
